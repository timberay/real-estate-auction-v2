# Decision: T2.6 Vitest 도입 — DEFERRED

**Date**: 2026-05-14
**Decider**: 자동 추천 적용 (master TODO T2.6 부분 완료 상태에서 마지막 결정)
**Status**: DEFERRED — 출시 후 1~2개월 운영 후 재평가
**Related**: 마스터 TODO T2.6, debt 감사 A3 (`docs/superpowers/plans/2026-05-12-existing-feature-debt.md` § A3)

## Context

`app/javascript/controllers/` 35개 Stimulus 컨트롤러에 unit test 0건. T2.6 "Stimulus JS 테스트 인프라"는
이미 부분 완료된 상태:

- ✅ 핵심 controller (profit_calculator / eviction / grade)는 기존 system test 로 cover
  (`test/system/profit_calculator_test.rb` 외)
- 🟡 Vitest 도입 — 본 결정으로 deferred

## Options

| Option | Pros | Cons |
|---|---|---|
| **A. Vitest 도입** | controller 단위 회귀 가드, 빠른 feedback loop, CI 시간 단축 | Node.js 의존 도입 (현 프로젝트는 importmap-only), CI 환경 새로 설정, ImportMap 환경에서 ES module 번들링 까다로움, jsdom + DOM stubs 기본 구성 부담 |
| **B. 현 system test 유지** | 추가 인프라 없음, 실 브라우저 동작 검증 | 컨트롤러 단위 회귀 미커버, Capybara 시동 비용으로 CI 느림 |
| **C. (자동 추천) DEFER 후 재평가** | 출시 직후 안정화에 집중, 1~2개월 운영 데이터로 결정 | 실제로 controller 회귀가 발생하면 그때 늦어짐 |

## Decision: C. DEFER 후 재평가

**근거**:

1. **현 system test 충분도가 검증되지 않음** — 실제로 controller 단위 회귀가 발생하는 빈도를
   1~2개월 운영 데이터로 측정해야 Vitest 도입 ROI를 판단 가능.
2. **인프라 비용** — ImportMap + Propshaft 만으로 Node.js 의존성 zero인 현 구조의 단순함이
   유지보수 비용을 크게 절감 중. 이를 깨뜨리는 결정은 명확한 회귀 패턴 확인 후에.
3. **현재 우선순위와의 trade-off** — 같은 노력으로 권리분석 정확도 / 매물 다양성 확장이
   사용자 가치에 더 직결.

## Re-evaluation Trigger

다음 중 하나라도 발생하면 즉시 Vitest 도입 검토:

- system test 가 catch 하지 못한 controller 단위 회귀가 1개월 내 2건 이상
- Capybara CI 시간이 5분 초과 (현재는 양호)
- 사용자 보고된 frontend 버그가 system test 우회 패턴 (예: 빠른 클릭, 비동기 race)을 띔
