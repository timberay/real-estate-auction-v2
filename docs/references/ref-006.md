명도 가이드 웹사이트의 IA(Information Architecture)를 페이지별 필수 구성요소로 정리합니다. 각 페이지는 단순 정보 전달이 아닌 **사용자의 의사결정을 돕는 도구** 관점으로 설계했습니다.

## 페이지 1. 명도 작업의 순서 (Process Timeline)

사용자가 "내가 지금 어느 단계에 있고, 다음에 뭘 해야 하는지"를 즉시 파악할 수 있어야 합니다.

### 필수 구성 섹션

| 섹션 | 목적 | 필수 포함 항목 |
|:---|:---|:---|
| **1-1. Hero / 개요** | 명도의 정의와 낙찰자 입장에서의 중요성 인지 | • 명도의 정의 (소유권 ≠ 점유권) <br>• 명도 미완료 시 리스크 (공실·시설훼손·수익 불가) <br>• 전체 소요기간 레인지 (협상 2~3개월 / 강제집행 3~6개월) |
| **1-2. 전체 타임라인 시각화** | 15단계(S1~S15) 한눈 파악 | • 가로형 또는 세로형 스텝 다이어그램 <br>• 각 단계 예상 소요일 <br>• 현재 진행 가능한 병행 작업 표시 (예: S5+S6 동시) <br>• 기한 알림 마커 (잔금납부 30일, 인도명령 6개월) |
| **1-3. 단계별 상세 카드** | 각 단계에서 해야 할 구체적 액션 | • 단계명 / 법적 근거 조문 <br>• "왜 이 단계가 필요한가" 1~2줄 설명 <br>• 필요 서류 목록 (체크박스) <br>• 예상 비용 <br>• 완료 판정 기준 (Yes 조건) <br>• 미완료 시 이동할 분기 안내 |
| **1-4. D-Day 계산기** | 가장 치명적인 6개월 기한을 놓치지 않도록 | • 잔금납부일 입력 → 인도명령 신청 마감일 자동 계산 <br>• 매각허가결정일 입력 → 즉시항고 마감일·잔금납부 마감일 계산 <br>• 계고일 입력 → 강제집행 가능일 계산 |
| **1-5. 비용 시뮬레이터** | 의사결정(강제집행 vs 이사비 합의) 지원 | • 전용면적 입력 → 강제집행 예납금 추정 (100~300만 원) <br>• 강제집행 예상 실비의 50~70% = 권장 이사비 제시 <br>• 공실 기회비용(월 임대료 × 예상 공실 월수) 비교 |
| **1-6. 다운로드 / 출력** | 오프라인 활용 | • 단계별 체크리스트 PDF <br>• 필요서류 리스트 PDF <br>• 개인별 타임라인 PDF (내 물건 기준) |

### 디자인 포인트
- **진행률 표시:** 사용자가 로그인해 자신의 물건 정보를 입력하면 "현재 S6 진행 중 (40%)" 같은 퍼센트 게이지
- **반복 강조 UI:** 인도명령 6개월 기한은 모든 단계 카드 상단에 고정 배너로 D-Day 표시

## 페이지 2. 명도 작업 시 발생하는 어려움과 해결 방안 (Troubleshooting)

사용자가 이미 문제 상황에 직면한 상태로 들어오는 페이지입니다. **빠른 검색과 즉시 적용 가능한 액션**이 핵심입니다.

### 필수 구성 섹션

| 섹션 | 목적 | 필수 포함 항목 |
|:---|:---|:---|
| **2-1. 문제 유형 선택 인터페이스** | 사용자가 자기 상황을 빠르게 식별 | • 카테고리 필터: 권리분석 리스크 / 점유자 문제 / 법적 절차 지연 / 집행 단계 문제 / 정산 분쟁 <br>• 상황 키워드 검색 (예: "연락두절", "유치권", "이사비 과다") <br>• "내 상황 진단" 질문 위저드 (3~5문항) → 해당 케이스 자동 매칭 |
| **2-2. 케이스별 상세 카드 (B1~B11)** | 각 분기 상황의 근본 대책 제공 | **필수 필드 구조:** <br>• 상황 요약 (1줄) <br>• 발생 원인 <br>• 법적 근거 (조문·판례 링크) <br>• 단계별 대응 액션 (번호화) <br>• 예상 추가 소요기간·비용 <br>• 주의사항 / 함정 <br>• 관련 서식 다운로드 <br>• 메인 플로우 복귀 지점 표시 |
| **2-3. 주요 리스크 Top 5 대책** | 가장 자주 발생하는 문제 집중 가이드 | • 유치권 신고 대응 (점유 검증·소멸시효·경매방해죄) <br>• 대항력 임차인 보증금 인수 <br>• 폐문부동·연락두절 <br>• 점유자 바꿔치기 <br>• 이사비 협상 결렬 |
| **2-4. 협상 템플릿 라이브러리** | 실전 문구·서식 제공 | • 내용증명 템플릿 (임차인용 / 채무자용 / 불법점유자용) <br>• 이사비 합의서 양식 <br>• 명도확인서 양식 <br>• 협상 대화 스크립트 (강경·중립·유화 3종) <br>• 경매방해죄 고소장 양식 |
| **2-5. 이사비 협상 계산기** | 합리적 이사비 범위 도출 | • 강제집행 실비 추정값 <br>• 예상 공실 기간 × 월 임대료 <br>• 시간가치 반영 NPV 계산 <br>• 권장 제시금액 / 상한선 자동 산출 |
| **2-6. 분기 흐름 시각화** | 현재 단계에서 분기 진입 시 전체 영향 파악 | • 메인 플로우 다이어그램 상에서 해당 분기가 어디에 연결되는지 하이라이트 <br>• 분기 진입 시 전체 일정 지연 예측 |
| **2-7. 실전 사례 아카이브** | 유사 사례 학습 | • 판례 기반 성공/실패 사례 <br>• 사용자 후기 (익명) <br>• 키워드 태그 기반 탐색 |

### 디자인 포인트
- **"지금 당장 해야 할 것" 우선 표시:** 각 케이스 카드 최상단에 "Today's Action" 박스 (예: "오늘 안에 점유이전금지가처분 신청")
- **법적 근거 접기/펼치기:** 법 조문은 접어두고 액션부터 노출, 필요 시 확장

## 페이지 3. 명도 관련 법률 지식 (Legal Knowledge Base)

### 디자인 포인트
- **인용 가능한 UI:** 각 조문·판례에 "인용하기" 버튼(클립보드 복사) — 사용자가 내용증명 작성 시 근거 활용
- **난이도 태그:** `초급/중급/고급` 표시로 사용자 수준별 필터링
- **상호 링크:** 3-1(조문) ↔ 3-3(판례) ↔ 페이지 2의 케이스 카드 양방향 링크

## 3개 페이지 공통 필수 요소

서비스 전반의 일관성과 신뢰성을 위한 글로벌 컴포넌트입니다.

| 요소 | 내용 |
|:---|:---|
| **면책 고지** | "본 정보는 일반 정보 제공 목적이며, 개별 사안은 변호사·법무사 상담 필요" (하단 고정) |
| **최종 업데이트일** | 각 문서·조문·판례 카드에 명시 (시의성 보증) |
| **출처 명시** | 인용 근거 링크 필수 (법제처·대법원·판례집) |
| **전문가 연결** | 페이지 하단 "이 케이스 변호사 상담 받기" CTA (제휴 기반 수익모델 연계 가능) |
| **내 물건에 적용** | 페이지 어디서든 "내 물건 분석에 추가" 버튼 → 개인 대시보드 연동 |
| **접근성** | 법률 용어 호버 시 툴팁 정의 표시 / 모바일 퍼스트 |

## 페이지 간 관계도

```
[페이지 1. 순서]          [페이지 2. 어려움]          [페이지 3. 법률지식]
  ├ 단계별 카드 ─────────► 해당 단계 분기 케이스
  └ D-Day 계산기            └ 케이스 상세 ──────────► 법조문·판례·서식
                                    ▲                        │
                                    └────────────────────────┘
                              용어·판례에서 역으로 실무 케이스 탐색
```

세 페이지가 서로 **양방향 링크**로 연결되어야, 사용자가 "지금 어느 단계 → 어떤 문제 → 법적 근거 → 서식 제출"을 끊김 없이 오갈 수 있습니다.

## 100항목 체크리스트 연동 제안

기존에 설계하신 100항목 체크리스트를 이 3페이지 구조와 매핑한다면:

- **페이지 1의 단계별 카드**에 체크리스크 항목을 끌어쓰기 (예: S1 권리분석 단계 → 체크리스트 12개 항목 표시)
- **페이지 2의 케이스 카드**에 "이 케이스를 예방하려면 체크할 항목" 3~5개 추천
- **페이지 3의 판례·조문**에 "이 법리를 활용하는 체크리스트 항목" 역링크

이렇게 하면 100항목 체크리스트가 **정적 리스트**가 아니라 **전체 서비스의 씨줄**이 되어 사용자의 학습·실행·회고를 관통하는 자산이 됩니다.

필요하시면 각 페이지의 ViewComponent 단위 설계(Rails 기준)나 와이어프레임 초안으로 발전시켜드릴 수 있습니다.
